## 주요 시나리오 (6가지)

### 시나리오 1: 정상 운영

| 항목 | 상태 |
|---|---|
| PC 서버 | ✅ 온라인 |
| RPi | ✅ 온라인 |
| 인터넷 | ✅ 연결 |
| FE 모드 | `online` (파란 헤더) |

**데이터 흐름:**
- 센서 → RPi Node-RED → PC 서버 API (`/api/sensors/collect`) + RPi SQLite 동시 저장
- FE → PC 서버 (port 3000) API로 데이터 조회

**제어 흐름:**
- FE → AWS API Gateway → Lambda → IoT Core → RPi MQTT 구독 → 디바이스 제어
- AWS 실패 시 → FE → RPi 로컬 제어 (`/api/control/local`) 폴백

**접근 가능 클라이언트:**

| 클라이언트 | 대시보드 | 데이터 소스 | 제어 |
|---|---|---|---|
| PC 브라우저 | ✅ | PC 서버 | AWS → 로컬 폴백 |
| 핸드폰 | ✅ | PC 서버 | AWS → 로컬 폴백 |
| RPi 터치패널 | ✅ (팜로컬 모드) | RPi SQLite | 로컬 직접 |

---

### 시나리오 2: PC 서버 다운 (RPi + 인터넷 정상)

| 항목 | 상태 |
|---|---|
| PC 서버 | ❌ 다운 |
| RPi | ✅ 온라인 |
| 인터넷 | ✅ 연결 |
| FE 모드 | `offline` (주황 헤더) |

**자동 전환 과정:**
1. `apiSwitcher.js` 10초 폴링 → PC `/health` 실패 감지
2. RPi 헬스체크 (`/api/system/mode`) → 성공
3. API 베이스 URL 자동 전환: `192.168.137.1:3000/api` → `192.168.137.30:1880/api`

**데이터 흐름:**
- 센서 → RPi SQLite에만 저장 (PC 전송 실패, `synced=0`으로 마킹)
- FE → RPi Node-RED API로 데이터 조회 (SQLite 기반)

**제어 흐름:**
- AWS IoT 경로 정상 작동 (인터넷 있음)
- 로컬 제어도 가능 (RPi 접근 가능)

**접근 가능 클라이언트:**

| 클라이언트 | 대시보드 | 데이터 소스 | 제어 | 비고 |
|---|---|---|---|---|
| PC 브라우저 | ✅ | RPi SQLite | AWS + 로컬 | apiSwitcher 자동 전환 |
| 핸드폰 | ⚠️ 제한적 | - | - | 아래 상세 설명 참조 |
| RPi 터치패널 | ✅ | RPi SQLite | 로컬 직접 | 팜로컬 모드 |

**핸드폰 접근 제한 상세:**
- PC 핫스팟이 살아있는 경우: 핸드폰 → RPi(192.168.137.30:1880) 접근 **가능**
  - 단, React 앱 자체를 어디서 로드하느냐가 문제 (PC 서버 다운 시 앱 서빙 불가)
  - 이미 앱이 로드되어 있다면 apiSwitcher가 RPi로 전환
- PC가 완전히 꺼진 경우: 핫스팟 소멸 → 핸드폰은 RPi에 접근 **불가**

**서버 복구 후:**
- apiSwitcher가 PC 서버 복구 감지 → 자동으로 PC 서버로 복원
- RPi SQLite의 `synced=0` 데이터 → 5분 간격 50건씩 PC 서버로 동기화

### 시나리오 5: 팜로컬 모드 (RPi 단독 운영)

| 항목 | 상태 |
|---|---|
| PC 서버 | 없음 (현장에 PC 없음) |
| RPi | ✅ 독립 운영 |
| 인터넷 | ❌ 없음 |
| FE 모드 | `farm-local` (초록 헤더) |

**설정 방법:**
- 설정 → 시스템 탭 → "팜로컬 모드" 토글 ON
- 또는 RPi에 `dist-farmlocal` 빌드 배포

**특징:**
- API 경로: `window.location.origin + '/api'` (같은 RPi 서버)
- 헬스체크 없음 (폴링 중지)
- 자동 로그인 (농장관리자, admin 권한)
- 메뉴 간소화: 대시보드 + 제어 + 설정 (3개만)
- Google Fonts 없이 시스템 폰트 사용

**접근 가능 클라이언트:**

| 클라이언트 | 대시보드 | 데이터 소스 | 제어 |
|---|---|---|---|
| RPi 터치패널 | ✅ | RPi SQLite | 로컬 직접 |
| 핸드폰 | ❌ (네트워크 없음) | - | - |
| PC 브라우저 | ❌ (네트워크 없음) | - | - |

**빌드 및 배포:**
```bash
# PC에서 빌드
npm run build:farmlocal

# RPi로 전송
scp -r dist-farmlocal/* lhk@192.168.137.30:/home/lhk/.node-red/static/

# RPi에서 Node-RED 재시작
pm2 restart node-red
```

## 추가 시나리오 (4가지)

### 시나리오 7: 간헐적 연결 끊김

PC 서버가 불안정하여 온/오프라인 반복되는 경우.

**현재 동작:**
- 10초 폴링으로 상태 전환 반복 → API 베이스 URL이 PC ↔ RPi 왔다갔다
- 진행 중인 API 호출이 실패할 수 있음

**대응:**
- 연속 N회 실패 후 전환하는 디바운싱 로직 추가 고려
- 현재는 1회 실패로 즉시 전환 (빠른 반응 우선)

### 시나리오 8: PC DB만 다운 (서버 프로세스 정상)

Express 서버는 살아있으나 PostgreSQL/TimescaleDB가 다운된 경우.

**현재 동작:**
- `/health` 엔드포인트 응답 정상 → apiSwitcher는 온라인으로 판단
- 실제 API 호출 (`/api/sensors/latest` 등) 시 500 에러 발생
- FE에서 개별 API 에러 처리

**대응:**
- `/health` 엔드포인트에 DB 연결 체크 추가 고려

### 시나리오 9: RPi Node-RED API 일부 미지원

RPi Node-RED REST API가 PC 서버의 모든 엔드포인트를 지원하지 않는 경우.

**현재 지원 상태:**

| 기능 | PC 서버 API | RPi Node-RED API | 오프라인 시 |
|---|---|---|---|
| 센서 최신 데이터 | ✅ `/api/sensors/latest` | ✅ `/api/sensors/latest` | 정상 |
| 센서 이력 | ✅ `/api/sensors/history` | ✅ `/api/sensors/history` | 정상 |
| 오늘 알림 수 | ✅ `/api/alerts/today/count` | ✅ `/api/alerts/today/count` | 정상 |
| 알림 목록 조회 | ✅ `/api/alerts` | ✅ `/api/alerts` | 정상 |
| 알림 확인/삭제 | ✅ `/api/alerts/:id` | ❌ | 불가 |
| 제어 이력 | ✅ `/api/control-logs` | ❌ | 불가 |
| 하우스 설정 조회 | ✅ `/api/config/farm/:id` | ⚠️ 글로벌 캐시 | 캐시된 값 |
| 하우스 설정 편집 | ✅ PUT `/api/config/...` | ❌ | 불가 |
| 자동화 규칙 | ✅ `/api/automation` | ❌ | 불가 |
| 시스템 설정 | ✅ `/api/config/system-settings` | ❌ | 불가 |
| 재배 일지 | ✅ `/api/journals` | ❌ | 불가 |

### 시나리오 10: AWS IoT만 불가

인터넷은 있으나 AWS 서비스에 문제가 있는 경우.

**현재 동작:**
- AWS 제어 API 호출 → 타임아웃/에러
- `controlApi.js`가 자동으로 로컬 제어 폴백 (`/api/control/local`)
- 센서 수집/데이터 조회에는 영향 없음

## 데이터 동기화

PC 서버가 복구되면 RPi SQLite에 쌓인 미전송 데이터(`synced=0`)가 자동 동기화됨:
- 주기: 5분 간격
- 배치: 50건씩 전송
- 성공 시: `synced=1`로 업데이트
- 보관 기간: 시스템 설정의 `retentionDays` (기본 60일, 매일 자정 정리)
